# 当前网站技术栈与架构说明

本文用于后续维护、项目交接与二次开发。

---

## 第一部分：网站技术栈

### 1. 总体架构

本项目不是传统意义上的“前后端分离双仓库”。

它采用的是：

- `Next.js App Router` 一体化架构
- 同一个仓库内同时包含：
  - 前台页面
  - 后台管理页面
  - 服务端接口
  - 数据库访问逻辑
  - 登录认证逻辑

可以把它理解为：

- 一个以 `Next.js` 为核心的全栈网站项目
- 适合内容型网站、学生社团官网、轻量后台管理场景

---

### 2. 前端技术栈

- `Next.js 15`
- `React 19`
- `TypeScript`
- 原生 `CSS`
- `next/image` 用于图片显示

前端特点：

- 前台与后台都写在 `src/app` 下
- 使用 React 组件拆分页面结构
- 样式集中在 `src/app/globals.css`

---

### 3. 后端技术栈

- `Next.js Server Actions`
- `Next.js Route Handlers`
- `NextAuth v5 beta`

后端职责主要包括：

- 管理后台表单提交
- 登录鉴权
- 上传文件访问
- 图片缩略图 / 水印处理

---

### 4. 数据库与 ORM

- `Prisma`
- `SQLite`

当前开发环境默认数据库：

- `prisma/dev.db`

线上建议：

- 数据库放在项目目录外的独立数据目录
- 例如：`/srv/data/zjuaaa-site/dev.db`

当前数据模型主要包括：

- `AdminUser`：管理员账号
- `SiteSetting`：站点设置
- `MediaAsset`：媒体库
- `KnowledgePost`：科普文章
- `ManualChapter`：天文手册
- `ActivityNotice`：活动
- `AstroPhoto`：天文摄影作品

---

### 5. 认证与权限

- 使用 `NextAuth`
- 登录方式为账号密码登录（`Credentials Provider`）
- 后台页面访问控制由两层实现：
  1. `middleware.ts` 做路由级拦截
  2. `src/lib/admin-session.ts` 做服务端兜底校验

特点：

- 未登录不能进入 `/admin`
- 已登录访问 `/admin/login` 会自动跳回后台

---

### 6. 图片与文件处理

当前网站的图片系统分为三层：

#### 6.1 原始上传文件

- 对外路径一般表现为 `/uploads/xxx`
- 实际文件目录由环境变量 `UPLOAD_DIR` 控制
- 开发环境可在本地目录
- 线上建议放到独立目录：
  - `/srv/data/zjuaaa-site/uploads`

#### 6.2 图片变体访问

网站新增了统一图片变体逻辑：

- `/media/...` 用于图片代理访问
- 支持：
  - `thumb`：缩略图
  - `original`：原图（可带水印）

用途：

- 前台默认加载缩略图，减轻首屏压力
- 原图访问可单独处理

#### 6.3 水印逻辑

当前原图查看支持水印处理。

目前水印内容为：

- `浙江大学学生天文爱好者协会 ZJUAAA`

### 7. Markdown / 手册渲染

天文手册页面使用：

- `react-markdown`

- `remark-math`
- `rehype-katex`
- `katex`

功能：

- 支持 Markdown

- 支持站内图片插入

---

### 8. 部署环境

推荐线上环境：

- `Ubuntu 22.04`
- `Node.js`
- `PM2`
- `Nginx`

典型职责分工：

## 第四部分：数据与文件流转说明

### 1. 图片上传流程

1. 后台媒体库上传图片
2. 图片写入上传目录
3. 数据库存入 `MediaAsset.filePath`
4. 前台或后台预览时优先走缩略图
5. 查看原图时走原图变体

### 2. 管理员登录流程

1. 访问 `/admin`
2. `middleware.ts` 先判断是否已登录
3. 页面服务端再通过 `admin-session.ts` 做二次校验
4. 登录成功后进入后台

### 3. 页面内容维护方式

- 站点基础信息：`SiteSetting`
- 科普文章：`KnowledgePost`
- 活动：`ActivityNotice`
- 摄影：`AstroPhoto`
- 手册：`ManualChapter`
- 媒体库：`MediaAsset`

## 第五部分：维护与交接时需要特别注意的点

### 1. 不要随意删除这几个文件

- `src/lib/auth.ts`
- `src/lib/admin-session.ts`
- `middleware.ts`
- `src/app/media/[...path]/route.ts`
- `prisma/schema.prisma`

这些都属于网站关键基础设施。

### 2. 图片系统已经不是“纯静态 public/uploads”

目前项目里：

- 既有 `/uploads/...`
- 也有 `/media/...`

维护时需要清楚：

- `/uploads/...` 更接近原始文件
- `/media/...` 是站内图片变体入口

### 3. 线上环境变量必须正确

尤其是：

- `DATABASE_URL`
- `UPLOAD_DIR`
- `AUTH_SECRET`
- `NEXTAUTH_SECRET`
- `NEXTAUTH_URL`

### 4. 登录问题优先检查这几项

- Nginx 是否正确转发 `Host` / `X-Forwarded-*`
- `NEXTAUTH_URL` 是否正确
- 数据库里管理员是否存在
- 是否重新执行过 `seed-admin.mjs`

### 5. 数据库目前是 SQLite

优点：

- 维护简单
- 成本低

限制：

- 不适合高并发
- 数据量很大后迁移成本会上升

如果未来访问量明显上升，可考虑迁移到 `PostgreSQL`。

- 小步修改

### 不建议的方式

## 结论

当前项目本质上是一个：

- 基于 `Next.js + Prisma + NextAuth + SQLite`
- 同仓库全栈实现
- 同时包含前台与后台
- 面向社团官网与轻量内容管理

的完整网站系统。

它的优点是：

- 易维护
- 易部署
- 适合交接
- 适合内容型站点

后续维护时，优先守住三条主线：

1. 登录鉴权不要乱动  
2. 图片路径与变体逻辑不要混淆  
3. 数据库结构修改前先看 `schema.prisma`
