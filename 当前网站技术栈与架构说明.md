# 当前网站技术栈与架构说明

这份文档是写给“没有代码背景，但需要知道这个网站现在是怎么搭起来的”同学看的。

---

## 1. 这个网站现在到底是什么架构

当前这个项目不是传统意义上的“前后端完全分离”。

它属于：

- `前后端一体化项目`
- 更准确地说，是 `Next.js 全栈项目`

意思是：

- 前台页面和后台管理界面都在同一个项目里
- 后端接口和数据处理逻辑也在同一个项目里
- 不是“前端一个仓库 + 后端一个仓库”那种拆开的结构

所以现在这个网站的运行方式可以理解成：

- `一个项目`
- 同时负责：
  - 前台官网
  - 后台管理
  - 登录鉴权
  - 数据读写
  - 图片上传

---

## 2. 现在用到的主要技术栈

### 前端

- `Next.js 15`
- `React 19`
- `TypeScript`
- 原生 `CSS`

前端负责：

- 首页
- 知识科普
- 社团活动
- 天文摄影
- 天文手册
- 关于我们
- 联系我们
- 加入我们

### 后端

- `Next.js App Router`
- `Server Actions`
- `NextAuth`

后端负责：

- 管理员登录
- 内容增删改查
- 图片上传
- 数据保存

### 数据库

- `Prisma`
- `SQLite`

目前数据库是一个本地文件数据库。

也就是说，现在网站的数据不是放在 MySQL / PostgreSQL 里，而是放在项目里的一个数据库文件里。

### 媒体与图片

- 图片上传后保存在：
  - `public/uploads`

### 进程与部署

上线时建议使用：

- `Node.js`
- `PM2`
- `Nginx`
- `Ubuntu 22.04`

---

## 3. 当前项目的核心工作方式

可以把它理解成下面这个结构：

```text
用户浏览器
    ↓
Next.js 网站
    ↓
后台逻辑 / 登录逻辑 / 数据处理
    ↓
SQLite 数据库文件
    ↓
public/uploads 图片目录
```

也就是说：

- 页面由 `Next.js` 渲染
- 管理员在后台编辑内容
- 内容被写入数据库
- 上传的图片保存在本地目录
- 前台再从数据库和图片目录里读取内容显示出来

---

## 4. 它是不是前后端分离

严格来说：

- `不是完全分离`

它不是这种结构：

```text
前端项目：Vue / React
后端项目：Java / Node / Python
数据库：单独服务
```

它更像：

```text
一个 Next.js 项目
前端页面 + 后端逻辑 + 管理后台 都在里面
```

所以它的优点是：

- 维护简单
- 部署简单
- 适合你们这种社团官网
- 不需要同时维护两个仓库

它的缺点是：

- 如果以后业务特别复杂，拆分空间会比纯分离式架构小一点
- 但对你们现在完全够用

- 摄影作品
- 手册章节

## 7. 现在这个项目要不要整个文件夹上传到 GitHub

不是“整个文件夹原样上传”，而是：

- 上传“源码和配置文件”
- 不上传“运行生成物、数据库、隐私文件、依赖包”

### 应该上传到 GitHub 的

通常应当上传：

- `src/`
- `prisma/schema.prisma`
- `scripts/`
- `public/` 里那些你们自己写死的静态资源
- `package.json`
- `package-lock.json`
- `tsconfig.json`
- `next.config.ts`
- 说明文档 `.md`

### 不应该上传到 GitHub 的

这些通常不要上传：

- `node_modules/`
- `.next/`
- `.env`
- `prisma/dev.db`
- `prisma/dev.db-shm`
- `prisma/dev.db-wal`
- `public/uploads/`

原因：

- `node_modules` 是依赖安装结果，太大了
- `.next` 是构建产物，部署时会重新生成
- `.env` 里有密钥和密码
- `dev.db` 是实际数据库，通常不直接公开上传
- `public/uploads` 是后台上传内容，通常也不直接放源码仓库

---

## 8. 如果上传 GitHub，最理想的做法是什么

建议你们把仓库理解成：

- `GitHub` 负责保存代码
- `服务器` 负责保存运行中的数据

也就是：

### GitHub 上保存

- 网站源码
- 样式
- 组件
- 页面
- 数据结构定义
- 部署文档

### 服务器上保存

- 真实数据库
- 后台上传图片
- `.env`

这才是更标准的方式。

---

## 9. 你们现在最需要知道的结论

可以直接记住这几句：

### 1. 这是一个前后端一体化项目

不是拆成两个仓库的前后端分离项目。

### 2. 前台、后台、登录、数据库读写都在同一个 Next.js 项目里

所以部署时只部署这一个项目。

### 3. 不是整个文件夹都上传 GitHub

要上传源码，但不要上传：

- `node_modules`
- `.next`
- `.env`
- `prisma/dev.db`
- `public/uploads`

### 4. 真正重要的数据不在代码里，在服务器里

最重要的是：

- 数据库文件
- 上传图片目录

---

## 10. 现在这个项目适不适合你们

非常适合当前阶段。

原因是：

- 你们没有代码基础
- 社团官网需求比较稳定
- 后台内容维护是重点
- 服务器也不会特别复杂

现在这套架构的优点是：

- 简单
- 稳定
- 易维护
- 上线成本低

对你们来说，比强行做前后端分离更合适。

---

## 11. 如果以后要升级，会怎么演进

未来如果内容越来越多，可以按这个方向升级：

### 第一阶段

现在这样：

- `Next.js + SQLite + 本地上传`

### 第二阶段

如果图片很多：

- 图片迁移到腾讯云 COS

### 第三阶段

如果访问量更大、多人同时维护：

- 数据库从 `SQLite` 换成 `PostgreSQL`

### 第四阶段

如果以后功能非常复杂：

- 再考虑把前后端拆开

但目前完全不需要直接走到这一步。

prisma/dev.db-shm

## 13. 我建议你们现在怎么做

最稳的做法是：

1. 先把源码传到 GitHub
2. 不把数据库和上传图片传上去
3. 服务器上单独运行这个项目
4. 服务器上单独保存图片和数据库
5. 以后更新时只更新代码，不覆盖数据库和上传图片

---

- `.gitignore`
- `GitHub上传说明.md`

这样你们就可以直接把项目整理后传仓库了。
