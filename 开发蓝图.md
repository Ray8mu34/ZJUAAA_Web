# 浙江大学学生天文爱好者协会网站开发蓝图

## 1. 蓝图目标

这份文档比《项目实施方案》更具体，目标是把项目拆到“可以直接开始开发”的程度。

重点覆盖：

- 项目整体技术结构
- 页面与组件拆分
- 后台功能与菜单设计
- 数据库表设计方向
- 图片与 `Markdown` 内容流
- 登录与多管理员方案
- 云部署建议
- 分阶段开发路线

## 2. 最终技术选型

结合你们目前的需求，推荐采用以下方案：

- 前端框架：`Next.js`
- 后端接口：`Next.js Route Handlers / Server Actions`
- 样式方案：`Tailwind CSS` + 自定义设计变量
- 数据库 ORM：`Prisma`
- 开发期数据库：`SQLite`
- 正式部署数据库：`PostgreSQL`
- 登录认证：`NextAuth/Auth.js` 凭证登录
- 内容编辑：`Markdown`
- 图片上传：本地上传接口，后续可切云存储
- 部署平台优先级：云平台优先，支持 `Vercel` / `Railway` / `Render` / 云服务器容器化部署

原因如下：

- `Next.js` 同时处理前台、后台、接口，结构最紧凑
- 你们更新频繁，`Prisma + PostgreSQL` 比纯文件系统更稳
- 账号密码登录与多管理员在这个方案中实现顺滑
- 图片、文章、活动等都适合统一走数据库驱动

## 9. Markdown 内容流设计

知识科普和天文手册都建议走统一的 `Markdown` 渲染流程。

### 9.1 后台编辑

后台表单中包含：

- 标题
- 摘要
- 作者
- 封面图
- `Markdown` 正文输入框

### 9.2 后端处理

保存时：

- 将 `Markdown` 原文直接存数据库
- 不在保存阶段转成 HTML

这样做的好处：

- 数据可迁移
- 修改渲染样式时不用重写内容
- 更利于后续版本升级

### 9.3 前端渲染

访问页面时：

- 读取数据库中的 `Markdown`
- 服务端渲染为 HTML
- 套用统一的文章样式组件

### 9.4 推荐支持的 Markdown 能力

第一版支持：

- 标题
- 段落
- 加粗 / 斜体
- 列表
- 链接
- 图片
- 分割线
- 引用

第二版可选：

- 表格
- 目录锚点
- 代码高亮
- 脚注

## 10. 图片流设计

### 10.1 上传流程

- 管理员在后台上传图片
- 图片保存到媒体目录
- 数据库写入图片信息
- 返回媒体资源 ID

### 10.2 绑定流程

- 编辑文章或页面时选择媒体库中的图片
- 将图片资源 ID 绑定到内容记录

### 10.3 占位图策略

设计稿中尚未提供的图片位先统一显示：

- 深色占位块
- 说明文字，例如“待上传图片”

这样前端结构可以一次性做完，后续只换资源。

### 10.4 云部署兼容

因为你们更偏向使用云服务，图片存储需要注意：

- 如果部署在纯静态或无持久磁盘的平台，本地上传目录不稳定
- 正式环境更推荐后续切换到对象存储

建议路线：

- 开发阶段：本地 `uploads`
- 上线阶段：迁移到云对象存储，例如兼容 S3 的方案

这样结构最稳。

## 11. 双语实现蓝图

### 11.1 必做范围

- 首页中英切换
- 页脚中英切换
- 公共导航中英切换

### 11.2 架构预留范围

- 所有核心内容模型预留 `Zh / En` 字段
- 前端读取时按当前语言优先取对应字段
- 若英文为空，则回退到中文

### 11.3 回退机制

例如英文页面访问时：

- 有英文内容则显示英文
- 无英文内容则显示中文内容

这样可以在未完成全站翻译前保持站点可用。

## 12. 云部署建议

你们提到更可能使用云服务，而不是自己的服务器，因此建议优先考虑以下路线。

### 12.1 推荐部署形态

- 前端与后台部署在同一云平台
- 数据库使用托管 `PostgreSQL`
- 图片使用对象存储

### 12.2 适合的云平台特征

优先考虑支持：

- `Next.js` 部署
- 环境变量配置
- 托管数据库接入
- 自定义域名
- HTTPS
- 持续部署

### 12.3 上线阶段建议

第一版建议采用：

- 应用部署到支持 `Next.js` 的云平台
- 数据库使用托管 `PostgreSQL`
- 图片先临时走平台磁盘或轻量存储

正式稳定后建议升级为：

- 应用平台
- 托管数据库
- 对象存储
- CDN

### 12.4 为什么不建议长期使用 SQLite 上线

- 多人使用时并发能力有限
- 云平台迁移不方便
- 备份与扩容能力不如托管数据库

因此：

- `SQLite` 适合本地开发
- `PostgreSQL` 适合正式上线
