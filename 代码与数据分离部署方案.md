# 代码与数据分离部署方案

这份文档对应你想要的线上目录结构：

```text
/srv/apps/
├── zjuaaa-site
├── future-project

/srv/data/
├── zjuaaa-site/
│   ├── dev.db
│   ├── uploads/
│   └── .env

/srv/backups/
├── zjuaaa-site/
```

这个方案比“代码和数据都放在同一个项目目录里”更适合长期维护。  
优点是：

- 代码更新更安全
- `git pull` 不会碰到数据库和上传图片
- 以后换服务器时迁移更清晰
- 备份更容易做

---

## 1. 现在这个项目能不能支持这种结构

可以，而且当前代码已经基本支持：

- 数据库路径由 `DATABASE_URL` 控制
- 上传目录由 `UPLOAD_DIR` 控制

也就是说：

- 代码可以放在 `/srv/apps/zjuaaa-site`
- 数据库可以放在 `/srv/data/zjuaaa-site/dev.db`
- 上传图片可以放在 `/srv/data/zjuaaa-site/uploads`

前端访问路径依旧保持：

```text
/uploads/文件名
```

不需要改后台使用方式。

---

## 2. 我推荐的最终目录

建议这样放：

```text
/srv/apps/zjuaaa-site
/srv/data/zjuaaa-site/dev.db
/srv/data/zjuaaa-site/uploads
/srv/data/zjuaaa-site/.env
/srv/backups/zjuaaa-site
```

这样最清楚：

- `/srv/apps/zjuaaa-site`：只有代码
- `/srv/data/zjuaaa-site`：只有运行数据
- `/srv/backups/zjuaaa-site`：只有备份

---

## 3. 为什么这样更好

### 代码目录

```text
/srv/apps/zjuaaa-site
```

这里适合放：

- `src`
- `public`
- `prisma/schema.prisma`
- `package.json`
- `next.config.ts`

也就是 GitHub 仓库内容。

### 数据目录

```text
/srv/data/zjuaaa-site
```

这里适合放：

- `dev.db`
- `uploads`
- `.env`

这些都不是源码，不应该和代码混在一起。

### 备份目录

```text
/srv/backups/zjuaaa-site
```

这里存：

- 数据库备份
- 上传图片备份

---

## 4. 首次部署目录创建

先在服务器创建目录：

```bash
mkdir -p /srv/apps
mkdir -p /srv/data/zjuaaa-site/uploads
mkdir -p /srv/backups/zjuaaa-site
```

然后把项目代码放到：

```bash
/srv/apps/zjuaaa-site
```

例如：

```bash
cd /srv/apps
git clone 你的仓库地址 zjuaaa-site
```

---

## 5. 环境变量应该怎么放

我建议把 `.env` 放在数据目录：

```text
/srv/data/zjuaaa-site/.env
```

内容建议写成：

```env
DATABASE_URL="file:/srv/data/zjuaaa-site/dev.db"
AUTH_SECRET="换成你自己的随机字符串"
AUTH_URL="https://你的域名"
ADMIN_USERNAME="admin"
ADMIN_PASSWORD="你的后台密码"
ADMIN_DISPLAY_NAME="ZJUAAA Admin"
NODE_ENV="production"
UPLOAD_DIR="/srv/data/zjuaaa-site/uploads"
```

这里最关键的是：

- `DATABASE_URL="file:/srv/data/zjuaaa-site/dev.db"`
- `UPLOAD_DIR="/srv/data/zjuaaa-site/uploads"`

---

## 6. 为什么我推荐把 .env 也放到 /srv/data

因为 `.env` 本质上不是源码，而是运行配置。

放到 `/srv/data/zjuaaa-site/.env` 的好处：

- 更新代码时不会覆盖
- 改配置时不容易误提交 GitHub
- 数据、上传目录、密钥都集中在一个地方

然后在代码目录里建立软链接：

```bash
ln -s /srv/data/zjuaaa-site/.env /srv/apps/zjuaaa-site/.env
```

这样项目仍然可以像平时一样读取 `.env`。

## 7. 首次部署完整流程

### 7.1 准备目录

```bash
mkdir -p /srv/apps
mkdir -p /srv/data/zjuaaa-site/uploads
mkdir -p /srv/backups/zjuaaa-site
```

### 7.2 下载代码

```bash
cd /srv/apps
git clone 你的仓库地址 zjuaaa-site
cd /srv/apps/zjuaaa-site
```

### 7.3 创建环境变量文件

```bash
nano /srv/data/zjuaaa-site/.env
```

写入：

```env
DATABASE_URL="file:/srv/data/zjuaaa-site/dev.db"
AUTH_SECRET="换成你自己的随机字符串"
AUTH_URL="https://你的域名"
ADMIN_USERNAME="admin"
ADMIN_PASSWORD="你的后台密码"
ADMIN_DISPLAY_NAME="ZJUAAA Admin"
NODE_ENV="production"
UPLOAD_DIR="/srv/data/zjuaaa-site/uploads"
```

### 7.4 在代码目录挂软链接

```bash
ln -s /srv/data/zjuaaa-site/.env /srv/apps/zjuaaa-site/.env
```

如果以后重建过目录，先删旧链接再重建：

```bash
rm -f /srv/apps/zjuaaa-site/.env
ln -s /srv/data/zjuaaa-site/.env /srv/apps/zjuaaa-site/.env
```

### 7.5 安装依赖

```bash
cd /srv/apps/zjuaaa-site
npm install
```

### 7.6 初始化数据库

```bash
npx prisma db push --skip-generate
npx prisma generate
npm run db:seed-admin
```

### 7.7 构建并启动

```bash
npm run build
pm2 start npm --name zjuaaa-site -- start
pm2 save
```

## 8. 如果你之前已经在线上跑过旧方案

如果原来数据库和图片还在代码目录里，需要迁移一次。

### 数据库迁移

如果旧数据库在：

```text
/srv/apps/zjuaaa-site/prisma/dev.db
```

迁移到新位置：

```bash
cp /srv/apps/zjuaaa-site/prisma/dev.db /srv/data/zjuaaa-site/dev.db
```

如果还有 WAL 文件，也一并处理：

```bash
cp /srv/apps/zjuaaa-site/prisma/dev.db-shm /srv/data/zjuaaa-site/ 2>/dev/null || true
cp /srv/apps/zjuaaa-site/prisma/dev.db-wal /srv/data/zjuaaa-site/ 2>/dev/null || true
```

### 上传图片迁移

如果旧图片在：

```text
/srv/apps/zjuaaa-site/public/uploads
```

迁移到新位置：

```bash
mkdir -p /srv/data/zjuaaa-site/uploads
cp -r /srv/apps/zjuaaa-site/public/uploads/* /srv/data/zjuaaa-site/uploads/ 2>/dev/null || true
```

## 9. 以后更新代码应该怎么做

以后更新就会很简单，因为数据已经和代码分开了。

### 标准更新流程

```bash
cd /srv/apps/zjuaaa-site
git pull
npm install
npx prisma db push --skip-generate
npx prisma generate
npm run build
pm2 restart zjuaaa-site
```

注意：

- 不用再碰数据库文件
- 不用再碰上传图片目录
- 不用担心 `git pull` 把数据覆盖

---

## 10. 备份应该怎么做

### 数据库备份

```bash
cp /srv/data/zjuaaa-site/dev.db /srv/backups/zjuaaa-site/dev-$(date +%F-%H%M).db
```

### 上传目录备份

```bash
rsync -a /srv/data/zjuaaa-site/uploads/ /srv/backups/zjuaaa-site/uploads-$(date +%F-%H%M)/
```

### 建议

- 每次大更新前备份一次数据库
- 每周备份一次数据库
- 每月备份一次上传图片

---

## 11. 这套结构下的 Nginx 和 PM2 要不要改

不用大改。

因为：

- 网站服务仍然跑在 `/srv/apps/zjuaaa-site`
- Nginx 仍然代理到 `127.0.0.1:3000`
- PM2 仍然启动 `next start`

真正变化的只有：

- `.env`
- 数据库位置
- 上传目录位置

---

## 12. 我推荐你现在就采用这套结构

原因很简单：

- 你们已经上线了
- 后续还会继续更新内容
- 有图片上传
- 有后台管理

这时候把“代码”和“运行数据”分开，是很值得做的一次整理。
